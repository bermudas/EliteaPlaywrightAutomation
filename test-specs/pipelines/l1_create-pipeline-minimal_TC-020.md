## Network Behavior
- `GET /api/v2/elitea_core/applications/prompt_lib/{ownerId}?tags=&sort_by=created_at&sort_order=desc&query=&agents_type=pipeline&limit=20&offset=0` — the Pipelines list endpoint (`agents_type=pipeline`, vs. Agents' `agents_type=classic`); fires on `/app/pipelines/all` mount, `200`.
- `POST /api/v2/elitea_core/applications/prompt_lib/{ownerId}` — fires on Save click, `201` on success. Response body includes `id` (357 in this run, used to build the redirect URL), `name` (persisted verbatim — this run's 28-char budgeted name was well under the 32-char cap and was NOT truncated; the response body already reflects the truncated value in cases that do exceed it, confirming the cap is enforced client-side before the request is even sent, same as Agents), `description`, `owner_id`, `versions[0]` (default `base` version scaffold with its own `id`, e.g. `382`), `meta.default_version_id`, `created_at`.
- `GET /api/v2/elitea_core/application/prompt_lib/{ownerId}/{id}` — fires once on landing on the detail page post-redirect, `200`, returns the full pipeline record for the edit form.
- `GET /api/v2/elitea_core/pipeline_trigger/prompt_lib/{ownerId}/pipeline/{versionId}/trigger` — fires on the detail page for the pipeline's own current version id (not a different pipeline — confirmed the `382` in this URL is this pipeline's own `versions[0].id`, not a cross-reference); `200`.
- `DELETE /api/v2/elitea_core/application/prompt_lib/{ownerId}/{id}` — fires on confirmed delete, `204` on success. No automatic redirect away from the (now-deleted) detail page URL was observed after this call — the page stayed put with the deleted entity still title-rendered in the tab. Automation should navigate to `/app/pipelines/all` explicitly after the `204` rather than waiting for an app-driven redirect. A subsequent fresh navigation to `/app/pipelines/all` correctly reflects the card's absence — standard lazy-load wait (per TC-004) still applies since it's a cold page load.
  - **[Implementer correction, Phase 4 Execute, 2026-07-02]**: this "no auto-redirect" observation, made during a `playwright-cli` cleanup pass, did NOT hold up under a real `npx playwright test` run — Pipelines DO redirect to `/app/pipelines/all` automatically after a successful delete, matching TC-023's own dedicated delete-case finding (which asserts an exact post-delete URL) rather than this AFS's incidental note. Confirmed live: an unconditional `page.goto('/app/pipelines/all')` issued immediately after the delete click raced the in-flight app-driven redirect and failed with `net::ERR_ABORTED` on 5/10 pipelines-module cases in this batch's first implementation pass. `tests/pages/entityForm.page.ts`'s `deleteEntity()` now waits for the natural redirect (falling back to an explicit navigation only if it doesn't materialize within 10s) — see that method's own doc comment. Automation implementers on this module should trust TC-023's dedicated finding over this AFS's own casual cleanup-pass observation for the redirect behavior specifically; the rest of this AFS's Network Behavior section is unaffected.
- Wait strategy for the implementer: `page.waitForResponse(resp => resp.url().includes('/applications/prompt_lib/') && resp.request().method() === 'POST' && resp.status() === 201)` after the Save click, and the equivalent `DELETE`/`204` wait after the confirmed-delete click — no fixed-duration sleep needed for either mutation.

## Known Defects Found During Exploration

- **[MINOR/DEFECT, pre-existing]** Pipeline Name field silently truncates at 32 characters, no counter or warning — same defect already filed against Agents as [`GH#27`](https://github.com/bermudas/EliteaPlaywrightAutomation/issues/27); **independently corroborated from this case** (comment added) confirming the identical `maxLength="32"` cap on the Pipeline create form's Name input. Not re-triggered here (this AFS's own test-data name was deliberately budgeted to 28 chars to avoid it) — but flagged in § Test Data that the case's own **literal** naming template (`TEST_Pipeline_Minimal_${timestamp}`, 35 chars) would NOT survive this cap as-authored, an implication worth flagging for the automation implementer even though the underlying defect itself isn't new.
- **[INFO/CLARIFICATION, pre-existing]** TC-020's Teardown step text ("Confirm deletion in modal dialog by clicking 'Confirm'") does not match the live product, which requires typing the pipeline's exact name into a text field before an initially-disabled "Delete" button enables — no button literally named "Confirm" exists in this flow. Same pattern already filed for Agents as [`GH#28`](https://github.com/bermudas/EliteaPlaywrightAutomation/issues/28), corroborated cross-module here via comment rather than re-filed. Reverse-masking guard applies: live behavior is a legitimate, safer UX pattern, not a defect — case text is stale.
- **[INFO, pre-existing]** Overflow-menu button `id="undefined-action"`, delete-confirmation dialog's broken `aria-labelledby="alert-dialog-title"` (resolves to nothing), and the confirm-name textbox's missing accessible name — all three already filed for Agents as [`GH#33`](https://github.com/bermudas/EliteaPlaywrightAutomation/issues/33), corroborated cross-module here via comment (confirms shared/generic dialog and toolbar infrastructure across Agents and Pipelines, not Agents-specific).
- **[INFO, pre-existing]** `getByRole('button', { name: 'Save' })` (without `exact: true`) strict-mode-violates against `Save As Version` on the pipeline **detail/edit** page — same shared-component pattern already filed for Agents as [`GH#34`](https://github.com/bermudas/EliteaPlaywrightAutomation/issues/34) (TC-012), already corroborated for Pipelines by a sibling analyst (TC-022) before this AFS was written. - **[INFO, new-but-minor, not filed separately]** The Pipelines list-view card face displays only the pipeline's **name** — the Description entered during creation is not visible anywhere on the collapsed card. The case's own Expected Final State states "Pipeline card displays the name and description provided," which does not hold for this view. Judged too low-severity/cosmetic to warrant its own ticket (same bar as several `[INFO]` case-text-drift items already on file for Agents); documented here per the reverse-masking guard rather than asserted as a false positive.

The core create-with-minimal-fields flow (Save disabled → Name alone → still disabled → +Description → enabled → Save → redirect → list membership → delete) behaves exactly as the case describes, modulo the items above — all of which are either pre-existing/corroborated or explicitly out-of-severity-bar clarifications.
