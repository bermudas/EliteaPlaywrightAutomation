### Must Generate (in test setup)
- 3 Skills, created via `${BASE_URL}app/skills/create`:
  1. Name `formatter`, Description "Formats text output..."
  2. Name `code-reviewer`, Description "Reviews code for quality issues..."
  3. Name `content-writer`, Description "Writes marketing/content copy..."
  - **Case-text drift (filed as `#129`, not a defect):** the case's own Test Data table specifies "Formatter", "Code Reviewer", "Content Writer" (mixed case, spaces). The live Skill-create form's "Name" field rejects this outright — confirmed live validation message: *"Name must be lowercase letters, digits and hyphens only (no spaces), and cannot start or end with a hyphen."* Typing "Formatter" produced `textbox [invalid]` with that exact message; typing `formatter` (lowercase, no space) was accepted and the Save button became enabled. The Skills list (both Card and Table view) displays the name **exactly as stored** — no title-casing transform is applied (`formatter` renders as literal lowercase `formatter` in both list views and on the detail page).
  - Because of this, the AFS below uses `formatter` / `code-reviewer` / `content-writer` as the three names throughout, and adapts the partial-search example accordingly (see next point).
- **Case-text drift on the "partial search term" (also part of `#129`):** the case's own example partial term is "Co" (2 characters), expected to match "Code Reviewer" and "Content Writer". Live behavior: typing fewer than 3 characters into the search box triggers a toast — *"The search key word should be at least 3 letters long"* — and produces no results/suggestions at all. With the case's original two-word intent no longer literally reproducible (no 3+ character substring is common to both `code-reviewer` and `content-writer`), this AFS substitutes **`ter`** as the partial-search term, which is a genuine 3-character substring common to `formatter` (`for-mat-ter`) and `content-writer` (`wri-ter`), and absent from `code-reviewer` — preserving the case's original intent (a partial term that matches exactly 2 of the 3 skills, excluding the third) with data that actually satisfies the live 3-character minimum.

## Actual Results (live system, this run — updated after 2026-07-10 re-verification)
- **While typing (no Enter)**: Skills list/table is not filtered yet. The search box opens a floating typeahead tooltip suggesting matching skill names (grouped under "Tags"/"Skills" headings), correctly implementing partial-match, exact-match, and no-match ("No Skills Match") suggestion filtering against the tooltip's own suggestion list. This matches the case's original expectation once the Enter step is included — **confirmed via this AFS's 2026-07-10 re-verification pass**, per human guidance on `#128`.

## Network Behavior
- **(2026-07-10 re-verification)** Confirmed via `browser_network_requests`: typing into the search box fires `GET /api/v2/elitea_core/search_options/prompt_lib/21?query=<term>&sort=id&order=desc&entities[]=tag&entities[]=skill&...` (200) — this is what backs the typeahead tooltip. Pressing **Enter** additionally fires `GET /api/v2/elitea_core/skills/prompt_lib/21?sort_by=created_at&sort_order=desc&query=<term>&tags=&limit=20&offset=0` (200) and `GET /api/v2/elitea_core/tags/prompt_lib/21?offset=0&limit=50&query=<term>&entity_coverage=skill` (200) — the `skills/prompt_lib` response is what drives the main list's re-render. Automation can wait on this specific request (`**/skills/prompt_lib/**?*query=*`) as a network-level confirmation of "filter applied," in addition to / instead of a DOM-only wait.
- Skill creation: `Save` triggers a create call that redirects to `/app/skills/all/{id}` on success (analogous to the Agents/Pipelines create-form pattern per `.agents/testing.md`) — exact endpoint path not captured this run, flagged as a gap for the implementer to fill via a `browser_network_requests` capture during automation.
